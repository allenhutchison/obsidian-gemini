A focused day on conversation-history efficiency. Two back-to-back PRs tackled the quadratic token-growth problem in long agent sessions: the first added a deterministic tool-result truncation pass, the second restructured it into a smarter two-phase pipeline that preserves Gemini's implicit prefix cache when the session is still below the compaction threshold. ## What shipped

### [#770 fix(agent): truncate old tool-result payloads in conversation history](https://github.com/allenhutchison/obsidian-gemini/pull/770)

Long agent sessions — especially coding sessions heavy with `read_file` calls — saw input-token counts grow roughly quadratically while output stayed flat. The culprit: large `functionResponse` payloads being replayed verbatim on every request. This PR adds an always-on truncation pass in `ContextManager.prepareHistory`: older tool-result turns whose serialized response exceeds 4 KB have their payload replaced with a compact elision marker, while the most recent two tool-result turns are left intact so the agent can still reason across its current batch. The pass runs before the compaction-threshold check, so many sessions stay under threshold and skip the more expensive summarization round-trip entirely.

### [#772 chore(deps): bump hono from 4.12.14 to 4.12.18](https://github.com/allenhutchison/obsidian-gemini/pull/772)

Dependabot security bump. `hono` 4.12.18 fixes three CVEs: cross-user cache leakage via `Vary: Authorization`/`Vary: Cookie` headers in the Cache middleware (GHSA-p77w-8qqv-26rm), CSS declaration injection via JSX SSR style objects (GHSA-qp7p-654g-cw7p), and improper `NumericDate` claim validation in JWT `verify()` (GHSA-hm8q-7f3q-5f36). None of those middleware surfaces are used by this plugin, but updating is the right call regardless.

### [#773 refactor(context): two-phase compaction (truncate first, summarize second)](https://github.com/allenhutchison/obsidian-gemini/pull/773)

Follow-up to #770 that fixes a cost regression introduced by always running the truncation pass. Truncating older history bytes invalidates Gemini's implicit prefix cache from the modification point forward — when the session is below the compaction threshold that cache-miss tax is pure waste. The refactored `prepareHistory` now uses a two-phase pipeline: below threshold, history is returned unchanged; once over threshold, phase 1 truncates old tool-result payloads and re-estimates the size; if phase 1 alone brings the session back under threshold, summarization is skipped entirely; otherwise phase 2 runs the existing summarization compaction on the already-truncated history. Documentation in `docs/reference/settings.md` was rewritten to explain the cache trade-off.
